## 📝 完整脚本

### 【0:00-0:30】开场 + 痛点

**PPT**: P1 封面 → P2 问题陈述

**动作**: 深呼吸，微笑，眼神接触评委

```
各位评委好，我是 TruthPress × SecondMe 的开发者。

（停顿 1 秒）

在开始之前，我想问大家一个问题：
你有没有遇到过这种情况——

在网上和人争论，说得再有理，也抵不过对方声音大？
或者争了半天，发现对方可能是水军？

（停顿 2 秒，让评委思考）

这就是我们发现的两个核心痛点：

第一，对手不真实。水军、机器人、回声室效应。
第二，观点不可信。情绪大于证据，结论无法结构化。

（指向 PPT）

所以，我们做了一个能让观点在证据面前重新排序的产品——
TruthPress × SecondMe 共识引擎。
```

**语速提示**: 
- 开场：正常语速，建立连接
- 痛点：放慢，强调关键词
- 产品名：清晰，停顿

---

### 【0:30-1:30】解决方案 + 演示

**PPT**: P3 解决方案 → P4 产品演示

**动作**: 切换 PPT，播放 Demo 视频（或现场演示）

```
我们的解决方案分为四步：

第一步，用户提交争议观点。比如"AI 应该取代教师"。

第二步，自动对手匹配。系统会优先寻找真实的历史反方观点。
如果找不到，DeepSeek 会生成一个可信的合成对手人格——
有名字、背景、立场，像真实网友一样。

（停顿 1 秒）

第三步，多智能体辩论。
这里不是开放聊天，而是线性流程：
Agent A 为用户观点辩护，
Agent B 代入对手人格反驳，
Judge C 负责抽取事实主张、联网搜索证据。

（指向 Demo 视频）

大家现在看到的是我们的"真理之梯"可视化系统。
每个方块代表一个观点：
蓝色是 Agent A 辩护，红色是 Agent B 反驳，
绿色表示验证通过，红色破碎表示被推翻。

（停顿 2 秒）

第四步，求真控制台。
Judge C 会输出结构化的裁决：
总体 verdict、每条主张的核查结果、证据强弱标记、证据来源。

这不是模糊的"我觉得你对"，
而是结构化的证据链。

谁更有理，不是靠气势，而是靠证据。
```

**语速提示**:
- 四步流程：清晰，每步之间停顿
- Demo 解说：配合视频节奏
- 金句：放慢，强调

---

### 【1:30-2:00】技术架构

**PPT**: P5 技术架构

**动作**: 指向架构图三个区域

```
技术架构方面，我们分为三层：

前端层，使用 Next.js 14、TypeScript、Framer Motion。
特别是真理之梯动画，基于 Apple HIG + Fluent 2 + Material 3 设计规范，
每个方块掉落、验证、破碎都有精确的物理参数。

AI 层，使用 DeepSeek，通过 OpenAI SDK 兼容。
三个 Agent 各司其职：
Agent A 辩护，Agent B 反驳，Judge C 裁决。

数据层，使用 Prisma + SQLite，集成 SecondMe OAuth。
支持实时 SSE 流式推送，让辩论过程同步到前端。

（停顿 1 秒）

整个系统，48 小时极限开发，1234 行代码，16 种动画效果。
```

**语速提示**:
- 技术名词：清晰，不疾不徐
- 数据：强调（48 小时、1234 行）

---

### 【2:00-2:30】核心亮点

**PPT**: P6 核心亮点

**动作**: 手势强调三个亮点

```
我们的核心亮点有三个：

第一，Agent 作为参与者，不是工具。
传统 AI 产品是你输入问题，AI 给你答案。
我们不是让 AI 替你思考，是让 AI 互相思考给你看。

（停顿 1 秒）

第二，强制事实核查。
Judge C 必须输出结构化的 verdict 和证据链。
每条主张都有证据强弱标记，来源可追溯。

（停顿 1 秒）

第三，辩论过程可视化。
真理之梯让 Agent 的思考过程可见、可理解、可信任。
这对于降低 Agent 幻觉、增强可信度非常重要。
```

**语速提示**:
- 每个亮点：清晰，停顿
- 差异化：强调"不是...而是..."

---

### 【2:30-2:50】竞品对比

**PPT**: P7 竞品对比

**动作**: 指向对比表格

```
和竞品相比，我们有四个差异化优势：

对手真实性：传统平台随机匹配，AI 聊天纯 AI，
我们是真人优先 + AI 兜底。

讨论结构：传统平台无序，AI 聊天开放，
我们是线性多智能体流程，不跑题。

事实核查：传统平台无，AI 聊天无，
我们 Judge C 强制输出。

证据展示：传统平台链接堆砌，AI 聊天文本引用，
我们是结构化证据链。

（停顿 1 秒）

简单说，我们不是再做一个聊天机器人，
而是做一个能让分歧被看见、被检验、被结构化总结的共识引擎。
```

**语速提示**:
- 对比：清晰，强调差异化
- 金句：放慢，有力

---

### 【2:50-3:00】总结 + Q&A

**PPT**: P8 总结

**动作**: 微笑，致谢

```
最后，用一句话总结：

我们把"有立场的社交讨论"，升级成了"有证据约束的共识生成过程"。

SecondMe 负责"人"，TruthPress 负责"真"。

让分歧被看见，让观点被检验，让共识被生成。

（停顿 1 秒）

演示地址：localhost:3000/cafe
GitHub：github.com/TruthPress_Hackathon

谢谢！欢迎提问。
```

**语速提示**:
- 总结：有力，每句停顿
- 联系方式：清晰
- 致谢：真诚

## 🎯 演示技巧

### 站位与肢体语言

```
站位：屏幕左侧 45°，不遮挡内容
手势：用手掌指向，不用手指
眼神：与每位评委保持 2-3 秒眼神接触
移动：每页 PPT 可小幅度移动，避免频繁走动
```

### 语速控制

```
正常语速：180 字/分钟
重点内容：放慢到 150 字/分钟
金句：120 字/分钟，停顿 2 秒
```

### 互动技巧

```
开场提问："你有没有遇到过..."（停顿 2 秒）
重点强调："这不是...而是..."（手势强调）
确认理解："这里清楚吗？"（眼神交流）
```

## ⚠️ 应急方案

### 如果 Demo 失败

```
话术：
"由于网络原因，我准备了录屏视频。
这是我们的产品演示..."

动作：
立即播放备用视频，继续解说
```

### 如果超时

```
可跳过部分：
- 技术架构（简化为 1 句话）
- 竞品对比（口头带过）

必须保留：
- 开场痛点
- 解决方案
- 核心亮点
- 总结
```

### 如果评委打断提问

```
策略：
1. 耐心听完问题
2. 简短回答（30 秒内）
3. "我会在后面的内容中详细讲到..."
4. 继续演示
```
